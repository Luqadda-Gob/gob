# Contributing to Gob

[English](CONTRIBUTING.md) | [Soomaali](CONTRIBUTING_SO.md)

Thank you for contributing to Gob. Gob is a programming language interpreter
written in Java. Contributions to the interpreter, language design,
documentation, tests, tooling, and release packaging are welcome.

## Before You Start

- Search the existing issues and pull requests before starting work.
- Open an issue before making a large change or changing language behavior.
- Keep pull requests focused on one problem or feature.
- Do not include generated build output, IDE files, or compiled `.class` files.

By contributing, you agree that your contribution will be licensed under the
Apache License 2.0 used by this repository.

## Development Requirements

Install the following tools:

- Git
- JDK 17 or newer
- Apache Maven

GraalVM is optional. It is only required when working on native executable
packaging.

Verify your environment:

```bash
java --version
javac --version
mvn --version
git --version
```

## Set Up the Project

Fork the repository, then clone your fork:

```bash
git clone https://github.com/YOUR-USERNAME/gob.git
cd gob
```

Create a branch for your change:

```bash
git switch -c feature/short-description
```

Use a descriptive branch name such as `fix/parser-error-reporting` or
`feature/list-methods`.

## Project Structure

```text
src/main/java/com/kq/gob/Gob/    Interpreter and language implementation
src/main/java/com/kq/gob/tools/  AST generation tools
src/test/java/                   Unit and integration tests
pom.xml                          Maven build configuration
```

The main entry point is:

```text
com.kq.gob.Gob.Gob
```

## Build and Run

Build the project:

```bash
mvn clean package
```

Run the command-line help:

```bash
java -jar target/gob-0.0.1.jar --caawi
```

Start the interactive prompt:

```bash
java -jar target/gob-0.0.1.jar --qor
```

Run a Gob source file:

```bash
java -jar target/gob-0.0.1.jar --file path/to/program.gob
```

## Run Tests

Run the complete test suite before submitting a pull request:

```bash
mvn test
```

Add or update tests when changing behavior:

- Scanner changes belong in `ScannerTest`.
- Parser changes belong in `ParserTest`.
- Interpreter changes belong in `InterpreterTest`.
- End-to-end language behavior belongs in `IntegrationTest`.

Tests should cover successful behavior and relevant error cases.

## Regenerate the AST Classes

`Expr.java` and `Stmt.java` are generated by `GenerateAst.java`. When changing
the supported expression or statement types, update the generator and
regenerate both files:

```bash
cd src/main/java
javac com/kq/gob/tools/GenerateAst.java
java com.kq.gob.tools.GenerateAst com/kq/gob/Gob
cd ../../..
```

Commit the updated generated files with the generator change. Remove any
generated `.class` files before committing.

## Coding Guidelines

- Follow the existing Java style and package structure.
- Use clear names that match the terminology already used by the interpreter.
- Keep methods focused and avoid unrelated refactoring.
- Preserve existing Gob language behavior unless the change is intentional and
  documented.
- Add comments only where the reason for the code is not apparent.
- Keep user-facing Gob messages consistent with the existing language.

## Native Executable Contributions

Native executable support should remain optional for contributors and users.
The standard Maven JAR must continue to build and run with a regular OpenJDK.

If you change GraalVM Native Image packaging:

- Document the required GraalVM version and platform tools.
- Verify that normal `mvn test` and `mvn package` workflows still work.
- Test the native executable on the target operating system.
- Do not commit generated executables to the repository.

Native executables are platform-specific, so Windows, Linux, and macOS builds
must be produced separately.

## Commit and Pull Request Guidelines

Before opening a pull request:

```bash
mvn clean test
mvn package
git status
```

Your pull request should:

- Explain the problem and the chosen solution.
- List any user-visible language changes.
- Include tests for changed behavior.
- Update documentation when commands or behavior change.
- Avoid unrelated formatting or refactoring changes.
- Confirm that tests and the Maven package build pass.

Use concise, descriptive commit messages. Examples:

```text
Fix parser handling of missing semicolons
Add tests for function return values
Document native executable builds
```

## Reporting Bugs

When reporting a bug, include:

- The Gob source code that reproduces the issue.
- The command used to run it.
- The expected result.
- The actual result and full error output.
- Your operating system, Java version, and Maven version.

For security-sensitive issues, do not publish exploit details in a public
issue.
